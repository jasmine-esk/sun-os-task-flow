Create-Task Flow — All States, Spec for Wayeez

Every state of creating a task — from the moment the user thinks "I need to add this" through the task settling into the plan. Two paths (quick + full), 4 entry points, Sunrun variants, and 7 edge cases. Wayeez should be able to design every state from this doc.

Owner: Wayeez (design) · Source: Jazz · Date: May 2026

What we're designing

The full lifecycle of creating a task in Sunline. Not the task component itself (that's already specced) — the journey of bringing a new task into existence, from trigger to commit.

This flow has to handle:

  • Two paths — quick (3 seconds, no AI) and full (30 seconds, with auto-spec)
  • Four entry points — inline + button, global + button, ⌘N keyboard, drag from backlog
  • A Sunrun variant — same flow + 2 extra fields
  • ~7 edge cases — no refs, errors, cancellations, validation, drafts, conversions
  • ~20 distinct states across all of the above
The big design idea: creating a task is not "fill out a form." It's a conversation between the user and the system. The system drafts; the user refines; both commit together. The flow has to feel collaborative, not transactional.

Total time budget for happy path: ~30s for full path with auto-spec; ~3s for quick path. Anything longer feels broken. Keep transitions short (100–300ms range), not long.

Open questions for Wayeez

  1. Modal vs side panel for full path? Current spec is centered modal. Alternative: slide-in side panel from right (matches existing task detail view). Which feels more natural for creation vs editing?
  2. How aggressive should auto-suggestion be during typing (F3)? Should there be a live suggestion popover as the user types ("did you mean…")? Or wait until F4? Recommend: wait. Live suggestions disrupt typing flow.
  3. "Approve all" shortcut? Power users will hate clicking ✓ on every row. Add a single "Approve all spec fields" button? Risk: defeats the purpose of review.
  4. Does the modal block Sunline interaction? Modal-style yes; side-panel-style no. Trade-off between focus and context. Recommend: backdrop dims but doesn't block — users can drag tasks in the background if they want, and the modal preserves state.
  5. How do drafts (E6) surface in the backlog? A new "Drafts" bucket? Or merged with "Deferred"? Recommend: separate "Drafts" bucket — these are works-in-progress, semantically different from deferred-but-defined tasks.
  6. Sunrun cost preview (S1) — show or hide? Showing creates cost-anxiety. Hiding obscures real spend. Recommend: show, but make it auto-collapse after first 5 missions when user has anchored on what's typical.
  7. Owner pre-fill behavior: when the user opens via global +, should we default to "no one" (force a pick) or to the user's own column (Jazz)? Recommend: default to last-used owner.
  8. What does "Save as draft" preserve? Only filled fields, or also the partial state (e.g., refs reviewed but not all)? Recommend: full state — if you reviewed 3 of 5 refs, drafts remembers that.

What this prototype simulates (vs. what's stubbed)

  1. Real: entry points (global + button, inline +, ⌘N keyboard), modal open/close animations, title input + 1.2s debounce, owner dropdown with Sunrun mid-flow conversion, refs keep/remove with kept-count, commit → task lands in column with glow, toast confirmation.
  2. Real: Sunrun config slide-in from owner-change, mission-type/inputs/review-mode/cost rendering, "Run mission now" button replacing "Create task," running-tag in column.
  3. Stubbed: auto-spec content is the same canned response regardless of title (would be LLM-generated against project context in production). Refs are fixed — would be Sunpull-pulled in production.
  4. Stubbed: field-by-field editing (description, mission type, est. time) — visually editable, not wired. The flow that matters for design review (entry → modal → spec → refs → commit) is fully wired.
  5. Not in this prototype: validation errors, network errors, save-as-draft, no-refs-found state. All documented in the Spec tab (E1–E7) with mock UI.
